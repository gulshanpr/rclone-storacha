# {{< icon "fas fa-cube" >}} Storacha

[Storacha](https://storacha.network/) is a decentralized storage network that provides 
data storage backed by Filecoin and IPFS. It offers content-addressed storage with 
cryptographic proofs and decentralized retrieval.

## Features

- Content-addressed storage using IPFS CIDs
- Permanent storage backed by Filecoin
- Decentralized and censorship-resistant
- Cryptographic proofs of storage

## Configuration

Storacha requires Node.js 18+ to be installed on your system, as it uses a Node.js 
bridge to communicate with the Storacha network.

### Prerequisites

1. Install Node.js 18 or later from [nodejs.org](https://nodejs.org/)
2. Ensure the `storacha-bridge.mjs` script is available in one of these locations:
   - Same directory as the rclone executable
   - Current working directory
   - `./backend/storacha/storacha-bridge.mjs`
   - `/usr/local/share/rclone/storacha-bridge.mjs`

### Standard options

Here are the Standard options specific to storacha (Storacha Decentralized Storage).

#### --storacha-space-did

Storacha space DID to operate on.

This is the decentralized identifier (DID) for your Storacha space.

Properties:

- Config:      space_did
- Env Var:     RCLONE_STORACHA_SPACE_DID
- Type:        string
- Required:    true

#### --storacha-email

Optional email address for Storacha authentication and account management.

Properties:

- Config:      email
- Env Var:     RCLONE_STORACHA_EMAIL
- Type:        string
- Required:    false

## Usage

Once configured, you can use your Storacha remote like any other rclone remote:

```console
# List files
rclone ls mystoracha:

# Copy a file to Storacha
rclone copy /path/to/file.txt mystoracha:

# Download a file from Storacha
rclone copy mystoracha:file.txt /local/path/

# Sync a directory
rclone sync /local/dir mystoracha:backup/
```

## Limitations

### Content Addressing

Storacha uses content-addressed storage (IPFS CIDs), which means:

- Files are identified by their cryptographic hash
- Modifying a file creates a new object with a different CID
- The backend handles the mapping between file paths and CIDs

### Node.js Requirement

This backend requires Node.js to be installed on your system. The backend 
communicates with a Node.js subprocess that handles the Storacha client operations.

### Modification Times

Modification times are supported with 1-second precision.

### Hashing

The backend does not expose content hashes through rclone's hash interface, 
as file identification is handled through IPFS CIDs internally.

## Technical Details

### Architecture

The Storacha backend uses a bridge architecture:

1. **Go Backend**: Implements the rclone filesystem interface
2. **Node.js Bridge**: Handles Storacha client operations
3. **IPC Communication**: JSON-based communication over stdin/stdout

### Storage Model

- Files are uploaded to Storacha and assigned IPFS CIDs
- The backend maintains a mapping between file paths and CIDs
- Storage is permanent and backed by Filecoin deals
- Files are retrievable through the IPFS network
